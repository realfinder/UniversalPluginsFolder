I included tests using FFT3DFilter, which I often use in my scripts.  This provides a data point for the effectiveness a very different multithreaded filter was able to achieve.


Data is from Table 1 above.


Multithreading is nice but it does not always make things faster.  The numbers in bold (see Table 1 above), give an indication of the maximum number of CPUs that can be effectively used depending on the number of shifts that have been specified.  It is nice that multithreading pays off most where you need it most.  With 64 shifts, you can more than double the frames per second whereas with 16 shifts you can barely get 1.5 times faster.  This is not surprising; the shifts loop is multithreaded with each shift assigned to one of the CPU threads.  For 64 shifts and 2 CPUs 32 shifts are sent to 1 thread and the other 32 are sent to the second thread.  The threading overhead is low and the relative amount of processing time spent in the shifts loop is high compared to the amount of processing time spent in the rest of SmoothD2.  When doing four shifts the case is reversed and the threading overhead causes a net loss when three or four CPUs are specified.  Even when specifying only 2 CPUs there is almost no speed increase.  The frame size does not matter.  When tested with 1080p material the exact same pattern was observed.

If multithreaded code is perfectly scalable then each additional CPU will increase the total frames per second by the fps of the first CPU.

The CPU effectiveness is the percentage of the perfect scalable increase in fps to the actual increase in fps.

The formula for the effectiveness of each additional CPU, expressed as a percentage, is

((currentValue - previousValue) / firstValue) * 100
